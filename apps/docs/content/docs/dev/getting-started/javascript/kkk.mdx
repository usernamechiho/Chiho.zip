# 코딩

22년에 코딩을 접한 이후로, 저는 코딩 테스트를 항상 피해다녔습니다. VSCode를 통해 HTML 태그를 작성하고, 브라우저에서 확인하며 無에서 有를 만들어내는 것이 재밌었기 때문에, 코딩 테스트는 굳이 엄두를 내며 도전하거나 공부하고 싶지 않았습니다. 알고리즘은 내가 발을 담구지 않아도 될 영역인 것 같았고, `프론트엔드 개발자는 코딩 테스트 준비 안 해도 된다는데?` 라는 말만 철썩같이 믿으며 코딩 테스트 준비를 피하기 급급했습니다.

물론 코딩 테스트는 필수가 아닙니다. 우리가 흔히 사용하던 **Array 메서드나 Object 메서드는 알고리즘이 추상화돼있는 함수**입니다. 저는 지금껏 개발을 하면서 알고리즘 지식에 막혀 개발이 안 되는 상황은 없었습니다. 하지만 내가 개발자로써 효율을 따지고 싶고, 공간 복잡도와 시간 복잡도를 저울질해보며 성능상 무엇이 더 나은 코드인가에 대한 확신을 가지고 싶다면, 코딩 테스트를 준비해야 합니다.

그렇기에 지금 와서, 다시 알고리즘을 공부해보려고 합니다. 코딩 테스트의 달인이 되려고 합니다. 제가 이렇게 코딩 테스트에 갑자기 오기가 생기는 이유는 아래와 같습니다.

1. 입사 전형 중 코딩 테스트가 있으면 입사 지원을 피하거나, 전형 진행을 포기하던 무능한 내가 떠오름
2. 효율 따지고 싶은데 방법을 몰라서 벙찌는 내가 싫음
3. 연차가 쌓이는데, 알고리즘에 약한 모습을 보이기 싫음
4. ~그냥 재밌으니까~ 개발자 스웩으로 코딩 테스트에 능한 개발자가 되고싶음

```js title="응애.js" showLineNumbers
function solution(d, budget) {
  // 정렬해서 for문 돌리면 될 것 같은데?
  const orderedArr = d.sort((a, b) => a - b);
  let money = 0;
  let arr = [];

  for (let i = 0; i < orderedArr.length; i++) {
    if (money + orderedArr[i] > budget) break;

    money = money + orderedArr[i];
    arr.push(orderedArr[i]);
  }

  return arr.length;
}
```

아직은 정말 비참한 정도의 수준이지만, 1개월, 6개월, 2년 뒤에 다시 이 글을 봤을 때 제가 분명 성장해있으리라 믿습니다. 하루 최소 한 문제는 꼭 풀겠습니다. 그 문제가 쉬운 문제든 어려운 문제든 일단 하루 한 문제 해결에 꾸준한 습관을 들이려고 합니다. [usernamechiho/algorithm](https://github.com/usernamechiho/algorithm)에 제 커밋이 끊기지 않게 할 테니, 기특한 마음으로 와서 마음껏 훈수 두시면 정말 감사드리겠습니다.
